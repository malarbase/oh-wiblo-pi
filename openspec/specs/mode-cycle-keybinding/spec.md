## ADDED Requirements

### Requirement: Mode cycle keybinding
The system SHALL register a keybinding with id `app.mode.cycle` that, when activated, advances the active agent mode through the ordered sequence `none → plan → ask → debug → none`. The default key for `app.mode.cycle` SHALL be `alt+m`. The prior `app.plan.toggle` keybinding id SHALL be removed; the legacy migration alias `togglePlanMode` in the keybinding migration map SHALL be updated to map to `app.mode.cycle` so existing user configurations are automatically migrated.

#### Scenario: Keybinding cycles from none to plan
- **WHEN** no special mode is active and the user presses the `app.mode.cycle` key
- **THEN** Plan mode is activated

#### Scenario: Keybinding cycles from plan to ask
- **WHEN** Plan mode is active and the user presses the `app.mode.cycle` key
- **THEN** Plan mode is exited and Ask mode is activated

#### Scenario: Keybinding cycles from ask to debug
- **WHEN** Ask mode is active and the user presses the `app.mode.cycle` key
- **THEN** Ask mode is exited and Debug mode is activated

#### Scenario: Keybinding cycles from debug back to none
- **WHEN** Debug mode is active and the user presses the `app.mode.cycle` key
- **THEN** Debug mode is exited and no special mode is active

#### Scenario: Legacy user config is migrated
- **WHEN** a user's `keybindings.json` contains an entry with key `togglePlanMode` or `app.plan.toggle`
- **THEN** that entry is automatically remapped to `app.mode.cycle` on load and a migration log warning is emitted

#### Scenario: Default key works on macOS
- **WHEN** the user presses `Alt+M` (Option+M) in a macOS terminal (Terminal.app, iTerm2, Ghostty) in either legacy ESC-prefix mode or Kitty protocol mode
- **THEN** the `app.mode.cycle` action fires

### Requirement: Hotkeys help table reflects mode cycle
The hotkeys help table generated by `buildHotkeysMarkdown()` SHALL replace the "Toggle plan mode" row with a "Cycle agent mode (none → plan → ask → debug)" row that displays the configured `app.mode.cycle` key.

#### Scenario: Hotkeys table shows cycle binding
- **WHEN** the user opens the hotkeys help
- **THEN** the table contains a row showing `app.mode.cycle` and its key, not a plan-toggle row

### Requirement: Unified agent_mode status segment
The session status bar SHALL display a single `agent_mode` segment that shows the currently active mode label (`Plan`, `Ask`, or `Debug`) when any special mode is active, and nothing when no mode is active. The three prior separate segments (`plan_mode`, `ask_mode`, `debug_mode`) SHALL be replaced by this single segment.

#### Scenario: Status bar shows Plan when plan is active
- **WHEN** Plan mode is active
- **THEN** the status bar `agent_mode` segment displays the plan mode label

#### Scenario: Status bar shows Ask when ask is active
- **WHEN** Ask mode is active
- **THEN** the status bar `agent_mode` segment displays the ask mode label

#### Scenario: Status bar shows Debug when debug is active
- **WHEN** Debug mode is active
- **THEN** the status bar `agent_mode` segment displays the debug mode label

#### Scenario: Status bar is empty when no mode is active
- **WHEN** no special mode is active
- **THEN** the `agent_mode` segment shows nothing / is hidden

### Requirement: Atomic mode transition via single helper
All mode transitions (entry and exit for any mode, triggered by slash command or keybinding) SHALL be routed through a single `#setActiveMode(next: ActiveMode)` method that atomically: (1) exits the currently active mode, (2) sets the authoritative `#activeMode` field, (3) enters the next mode, (4) updates the status bar, (5) calls `appendModeChange`. No mode-transition code path SHALL update the status bar or call `appendModeChange` outside of this helper.

#### Scenario: No partial state after mode switch
- **WHEN** the user switches from Plan mode to Ask mode via any mechanism (slash command or keybinding)
- **THEN** the status bar shows Ask (not Plan) and `appendModeChange("ask")` has been called exactly once; Plan mode's context is removed

#### Scenario: Simultaneous mode prompts cannot appear
- **WHEN** a mode transition completes
- **THEN** at most one mode's context block is present in the session's message context
